# Superpowers 泛用式 AI 任务处理框架 🚀

> 基于 TDD 和系统化思考的完整复杂问题解决工作流

**v2.0 重大更新**：从开发专用拓展为泛用式任务处理框架！

---

## 📖 简介

Superpowers 是一个强大的 AI 任务处理框架，将你的 AI 代理转变为真正具备问题解决能力的伙伴。它基于严格的测试驱动开发（TDD）实践和系统化的思考方法论，提供从需求分析到代码实现、从信息收集到方案决策的全流程自动化。

**v2.0 核心升级**：
- ✅ 从开发专用拓展为泛用式框架
- ✅ 新增任务复杂度评估机制
- ✅ 新增 4 个泛用式任务处理流程
- ✅ 新增 Token 效率优化策略
- ✅ 适用于软件开发、信息查询、问题分析等各种复杂任务

---

## 🎯 核心理念

### 五大核心原则

1. **测试驱动开发 (TDD)** - 没有测试的代码都是假的（开发任务）
2. **系统性方法** - 告别临时性的思考，系统化拆解复杂问题
3. **复杂性降低** - 简单是首要目标，复杂问题拆解为简单步骤
4. **证据优于声明** - 验证后再说成功
5. **效率优先** - 显著提升 token 调用效率和任务效果

## 🌟 泛用式技能（v2.0 新增）

### 通用头脑风暴 (General Brainstorming) 🧠

**适用场景**：需要深度思考、设计、规划、分析的复杂任务

**核心流程**：
1. 理解任务 - 明确目标、约束、成功标准
2. 澄清问题 - 一次一个问题，确保理解准确
3. 探索方案 - 提出 2-3 种方法，讨论权衡
4. 分块展示 - 将复杂结果分块展示，每块获取确认
5. 结构化输出 - 生成结构化的文档或报告
6. 验证确认 - 确认结果符合需求

**输出类型**：
- 设计文档（`docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`）
- 分析报告（`docs/superpowers/reports/YYYY-MM-DD-<topic>-analysis.md`）
- 实施计划（`docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`）

---

### 任务拆解与执行 (Task Decomposition) 📝

**适用场景**：需要多步骤执行的任务

**核心流程**：
1. 任务拆解 - 将复杂任务拆解为 2-5 分钟的原子任务
2. 优先级排序 - 按依赖关系和重要性排序
3. 分步执行 - 逐步执行每个任务，每步验证
4. 进度汇报 - 定期汇报进度，获取反馈
5. 结果汇总 - 汇总所有子任务结果

---

### 信息收集与综合 (Information Gathering) 🔍

**适用场景**：需要从多个信息源收集和综合信息的任务

**核心流程**：
1. 信息需求分析 - 明确需要什么信息、为什么需要
2. 信息源识别 - 确定最佳信息源（网络搜索、文件读取、API 调用等）
3. 并行收集 - 尽可能并行收集信息以节省 token
4. 信息综合 - 将收集的信息综合、去重、验证
5. 结构化输出 - 生成结构化的报告或文档

**效率优化**：
- 一次性收集所有需要的信息，避免多次往返
- 使用批量操作代替循环操作
- 优先使用本地资源（文件、缓存），再使用网络资源

---

### 方案设计与决策 (Solution Design) 💡

**适用场景**：需要设计解决方案或做决策的任务

**核心流程**：
1. 需求分析 - 深入理解问题本质和约束条件
2. 方案生成 - 生成 2-3 个可行方案
3. 方案评估 - 从多个维度评估每个方案（成本、收益、风险、可行性）
4. 推荐方案 - 基于评估推荐最佳方案，并解释理由
5. 实施计划 - 为推荐方案制定实施计划（如需要）

**评估维度**：
- 技术可行性
- 成本（时间、资源、金钱）
- 风险（技术风险、业务风险）
- 可维护性/可扩展性
- 用户体验/满意度

## 🚨 任务复杂度评估

### 什么是复杂任务？

**复杂任务特征**：
- 需要多步推理或多轮信息收集
- 涉及多个信息源或工具调用
- 需要创造性思维或方案设计
- 涉及权衡决策或不确定性
- 需要结构化输出（报告、文档、计划）
- 单次回复无法充分覆盖

### 什么是简单任务？

**简单任务特征**：
- 单一信息查询（如："现在几点了"、"北京天气"）
- 单个文件操作（如："读取这个文件"）
- 直接答案问题（如："1+1等于几"）
- 单步操作（如："发送这个消息"）

### 自动触发规则

**必须触发 Superpowers 的场景**：
- ✅ "帮我创建一个..."（新功能、新文档、新计划）
- ✅ "帮我分析..."（复杂分析、深度调研）
- ✅ "帮我设计..."（系统设计、方案设计）
- ✅ "帮我优化..."（性能优化、流程优化）
- ✅ "帮我解决..."（复杂问题、Bug 修复）
- ✅ "帮我制定..."（计划、策略、方案）
- ✅ 涉及多个步骤或子任务
- ✅ 涉及不确定性或权衡决策

**不需要触发 Superpowers 的场景**：
- ❌ 单一信息查询
- ❌ 简单文件操作
- ❌ 直接答案问题
- ❌ 单步操作
- ❌ 已有明确路径的常规任务

## ⚡ Token 效率优化

### 核心原则

1. **批量操作原则** - 一次性收集所有需要的信息
2. **并行处理原则** - 并行处理独立的子任务
3. **结构化输出原则** - 一次性返回完整结果
4. **缓存利用原则** - 优先使用已有信息
5. **工具选择优化** - 选择最直接的工具完成任务

### 效率对比

**❌ 低效做法**：
```
用户：帮我查 A、B、C 三件事
AI：查 A... 回复
用户：继续
AI：查 B... 回复
用户：继续
AI：查 C... 回复
```

**✅ 高效做法**：
```
用户：帮我查 A、B、C 三件事
AI：一次性收集 A、B、C 三件事的信息...
AI：[一次性返回所有结果]
```

## 📚 开发技能详解（v1.0）

### 1. Brainstorming（头脑风暴）💡

**触发条件**：任何创意工作之前

**核心流程**：
- 探索项目上下文
- 询问澄清问题（一次一个）
- 提出 2-3 种方案及权衡
- 展示设计（分块展示，每个部分都需要批准）
- 编写设计文档
- 规范审查循环
- 用户审查

**输出**：`docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`

---

### 2. Writing Plans（编写计划）📝

**触发条件**：在拥有规范或需求的多步任务之前

**核心流程**：
- 范围检查
- 文件结构映射
- 编写细粒度任务（每个步骤 2-5 分钟）
- 计划文档审查循环

**输出**：`docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`

**任务粒度示例**：
- "编写失败的测试" - 步骤
- "运行它以确保失败" - 步骤
- "编写最小代码使其通过" - 步骤
- "运行测试并确保通过" - 步骤
- "提交" - 步骤

---

### 3. Test-Driven Development（测试驱动开发）🧪

**触发条件**：实现任何功能或错误修复之前

**铁律**：
```
没有失败的测试，就没有生产代码
```

**RED-GREEN-REFACTOR 循环**：

#### RED - 编写失败的测试
- 编写一个显示应该发生什么的测试
- 要求：一种行为，清晰的名称，真实代码

#### 验证 RED - 看它失败
- **强制执行。不要跳过。**
- 确认测试失败（不是错误）
- 确认失败消息符合预期

#### GREEN - 最小代码
- 编写最简单的代码来通过测试
- 不添加功能，不重构其他代码

#### 验证 GREEN - 看它通过
- **强制执行。**
- 确认测试通过
- 确认其他测试仍然通过

#### REFACTOR - 清理
- 只在绿色之后
- 删除重复，改进名称，提取助手
- 保持测试绿色，不添加行为

---

### 4. Requesting Code Review（请求代码审查）👀

**触发条件**：任务之间，自动激活

**核心流程**：
- 审查计划
- 按严重程度报告问题
- 关键问题阻止进展

**审查重点**：
- 是否符合规范？
- 代码质量如何？
- 测试是否充分？
- 是否遵循了 DRY、YAGNI 原则？

---

### 5. Subagent-Driven Development（子代理驱动开发）🤖

**触发条件**：拥有计划，如果框架支持子代理

**核心流程**：
- 每个任务使用新的子代理
- 两阶段审查（规范符合性，然后代码质量）
- 检查和继续

**优势**：
- Claude 可以自主工作数小时而不偏离计划
- 每个子代理都是"热情的初级工程师"
- 完全隔离的工作空间

---

### 6. Using Git Worktrees（使用 Git 工作树）🌳

**触发条件**：设计批准后，在实施之前

**核心流程**：
- 创建隔离的工作空间在新分支
- 运行项目设置
- 验证干净的测试基线

---

### 7. Finishing a Development Branch（完成开发分支）✅

**触发条件**：任务完成时

**核心流程**：
- 验证测试
- 展示选项（合并/PR/保留/丢弃）
- 清理工作树

## ⚠️ 常见陷阱

### 反模式："这太简单了，不需要设计"

每个项目都经历这个过程。待办事项列表、单功能实用程序、配置更改——所有这些。"简单"的项目是未检查的假设造成最多浪费的地方。设计可以很短（对于真正简单的项目只有几句话），但你**必须**展示它并获得批准。

---

## 🚨 常见错误（以及为什么它们是错误的）

| 理由 | 现实 |
|------|------|
| "太简单了，不需要测试" | 简单的代码也会破坏。测试需要 30 秒。 |
| "我会在之后测试" | 立即通过的测试证明不了任何东西。 |
| "之后测试达到相同目标" | 测试后 = "这有什么作用？" 测试前 = "这应该有什么作用？" |
| "我已经手动测试了所有边缘情况" | 临时 ≠ 系统性。没有记录，无法重新运行。 |
| "删除 X 小时的工作是浪费的" | 沉没成本谬误。保留未验证的代码是技术债务。 |

## 🎓 最佳实践

### DO（应该做）✅

- 每个项目都通过设计流程（即使是简单的）
- 一次问一个问题
- 提出多种方案并讨论权衡
- 编写细粒度任务（2-5 分钟）
- TDD：先写测试，看它失败，编写最小代码
- 在计划中包含确切的文件路径、完整代码、验证步骤
- 频繁提交
- 代码审查在任务之间

### DON'T（不应该做）❌

- 在没有设计的情况下实现功能
- 一次问多个问题
- 跳过 TDD（即使是"简单"项目）
- 在测试之前编写代码（删除它，重新开始）
- 添加不必要的功能（YAGNI）
- 编写大而复杂的任务
- 跳过审查循环

- **重大更新**：从开发专用拓展为泛用式任务处理框架
- 新增任务复杂度评估机制
- 新增 4 个泛用式任务处理流程：
  - 通用头脑风暴 (General Brainstorming)
  - 任务拆解与执行 (Task Decomposition)
  - 信息收集与综合 (Information Gathering)
  - 方案设计与决策 (Solution Design)
- 新增 Token 效率优化策略
- 新增全局认知指导
- 更新技能索引和文档结构

## 💎 核心价值主张

Superpowers v2.0 不仅仅是软件开发框架，它是一个**通用的复杂问题解决引擎**：

- **对开发者**：严格的工程方法论，从设计到实现的全流程自动化
- **对研究员**：系统化的信息收集和综合，高效的研究流程
- **对决策者**：结构化的方案设计和评估，清晰的决策支持
- **对所有用户**：智能的任务拆解和执行，显著的 token 效率提升

**核心理念**：
> 任何复杂任务都可以通过系统化思考和任务拆解变得简单。
> 让 AI 代理不只是工具，而是真正的问题解决伙伴。
