# Superpowers - 泛用式 AI 任务处理框架 🚀

Superpowers 是一个完整的 AI 任务处理框架，基于可组合的"技能"和严格的系统化方法论，让你的 AI 代理具备真正的问题解决能力。

**v2.0 重大更新**：从开发专用拓展为泛用式任务处理框架！

---

## 核心理念 🎯

### 五大核心原则

1. **测试驱动开发 (TDD)** - 没有测试的代码都是假的（开发任务）
2. **系统性方法** - 告别临时性的思考，系统化拆解复杂问题
3. **复杂性降低** - 简单是首要目标，复杂问题拆解为简单步骤
4. **证据优于声明** - 验证后再说成功
5. **效率优先** - 显著提升 token 调用效率和任务效果

### 泛用式工作流架构

```
用户需求 → 任务复杂度评估 →
  ├─ 简单任务 → 直接执行 → 完成
  └─ 复杂任务 → 头脑风暴 → 任务拆解 → 分步执行 → 验证 → 完成
```

## 🎯 任务复杂度评估

### 什么是复杂任务？

**复杂任务特征**：
- 需要多步推理或多轮信息收集
- 涉及多个信息源或工具调用
- 需要创造性思维或方案设计
- 涉及权衡决策或不确定性
- 需要结构化输出（报告、文档、计划）
- 单次回复无法充分覆盖

### 什么是简单任务？

**简单任务特征**：
- 单一信息查询（如："现在几点了"、"北京天气"）
- 单个文件操作（如："读取这个文件"）
- 直接答案问题（如："1+1等于几"）
- 单步操作（如："发送这个消息"）

### 评估检查清单

**任务复杂度自检**：

- [ ] 需要多步思考或推理？
- [ ] 需要多个工具调用？
- [ ] 需要设计或规划？
- [ ] 需要权衡多个方案？
- [ ] 需要结构化输出？
- [ ] 需要验证或确认？

**如果任何一个答案是"是"，则是复杂任务，应启动 Superpowers 流程。**

### 自动触发规则

**必须触发 Superpowers 的场景**：
- ✅ "帮我创建一个..."（新功能、新文档、新计划）
- ✅ "帮我分析..."（复杂分析、深度调研）
- ✅ "帮我设计..."（系统设计、方案设计）
- ✅ "帮我优化..."（性能优化、流程优化）
- ✅ "帮我解决..."（复杂问题、Bug 修复）
- ✅ "帮我制定..."（计划、策略、方案）
- ✅ 涉及多个步骤或子任务
- ✅ 涉及不确定性或权衡决策

**不需要触发 Superpowers 的场景**：
- ❌ 单一信息查询
- ❌ 简单文件操作
- ❌ 直接答案问题
- ❌ 单步操作
- ❌ 已有明确路径的常规任务

## 🌟 泛用式任务处理流程

### 流程 1：通用头脑风暴 🧠

**适用场景**：任何需要深度思考、设计、规划、分析的复杂任务

**核心流程**：
1. **理解任务** - 明确目标、约束、成功标准
2. **澄清问题** - 一次一个问题，确保理解准确
3. **探索方案** - 提出 2-3 种方法，讨论权衡
4. **分块展示** - 将复杂结果分块展示，每块获取确认
5. **结构化输出** - 生成结构化的文档或报告
6. **验证确认** - 确认结果符合需求

**输出类型**：
- 设计文档（如 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`）
- 分析报告（如 `docs/superpowers/reports/YYYY-MM-DD-<topic>-analysis.md`）
- 实施计划（如 `docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`）

---

### 流程 2：任务拆解与执行 📝

**适用场景**：需要多步骤执行的任务

**核心流程**：
1. **任务拆解** - 将复杂任务拆解为 2-5 分钟的原子任务
2. **优先级排序** - 按依赖关系和重要性排序
3. **分步执行** - 逐步执行每个任务，每步验证
4. **进度汇报** - 定期汇报进度，获取反馈
5. **结果汇总** - 汇总所有子任务结果

**任务粒度**：
- 每个 atomic task 应该 2-5 分钟可完成
- 每个任务有明确的输入和输出
- 任务之间依赖关系清晰
- 可以并行执行的任务要标注

---

### 流程 3：信息收集与综合 🔍

**适用场景**：需要从多个信息源收集和综合信息的任务

**核心流程**：
1. **信息需求分析** - 明确需要什么信息、为什么需要
2. **信息源识别** - 确定最佳信息源（网络搜索、文件读取、API 调用等）
3. **并行收集** - 尽可能并行收集信息以节省 token
4. **信息综合** - 将收集的信息综合、去重、验证
5. **结构化输出** - 生成结构化的报告或文档

**效率优化**：
- 一次性收集所有需要的信息，避免多次往返
- 使用批量操作代替循环操作
- 优先使用本地资源（文件、缓存），再使用网络资源
- 合理利用子代理并行处理

---

### 流程 4：方案设计与决策 💡

**适用场景**：需要设计解决方案或做决策的任务

**核心流程**：
1. **需求分析** - 深入理解问题本质和约束条件
2. **方案生成** - 生成 2-3 个可行方案
3. **方案评估** - 从多个维度评估每个方案（成本、收益、风险、可行性）
4. **推荐方案** - 基于评估推荐最佳方案，并解释理由
5. **实施计划** - 为推荐方案制定实施计划（如需要）

**评估维度**：
- 技术可行性
- 成本（时间、资源、金钱）
- 风险（技术风险、业务风险）
- 可维护性/可扩展性
- 用户体验/满意度

## 📊 Token 效率优化策略

### 1. 批量操作原则

**❌ 低效做法**：
```
用户：帮我查 A、B、C 三件事
AI：查 A... 回复
用户：继续
AI：查 B... 回复
用户：继续
AI：查 C... 回复
```

**✅ 高效做法**：
```
用户：帮我查 A、B、C 三件事
AI：一次性收集 A、B、C 三件事的信息...
AI：[一次性返回所有结果]
```

### 2. 并行处理原则

当有多个独立任务时，标记为可并行，然后并行处理：

**可并行任务**：
- 多个独立的信息查询
- 多个独立的文件操作
- 多个独立的 API 调用

**不可并行任务**：
- 有依赖关系的任务
- 需要用户确认的任务
- 需要前序任务结果的任务

### 3. 结构化输出原则

一次性返回完整结果，而不是逐步提示：

**❌ 低效**：
```
AI：我查到了一些信息，要不要继续？
用户：继续
AI：还有一些信息...
用户：继续
```

**✅ 高效**：
```
AI：[返回完整的结构化报告，包含所有信息]
AI：需要我补充什么吗？
```

### 4. 缓存利用原则

- 优先使用已有信息（memory、文件、缓存）
- 避免重复查询相同信息
- 记录重要信息到 memory 文件

### 5. 工具选择优化

- 选择最直接的工具完成任务
- 避免不必要的中间步骤
- 一个工具能完成的，不要用多个工具组合

## 技能体系 🛠️

### 核心技能链

#### 1. Brainstorming（头脑风暴）💡
**触发条件**：任何创意工作之前（创建功能、构建组件、添加功能、修改行为）

**核心流程**：
- 探索项目上下文
- 询问澄清问题（一次一个）
- 提出 2-3 种方案及权衡
- 展示设计（分块展示，每个部分都需要批准）
- 编写设计文档
- 规范审查循环
- 用户审查
- 转移到实现计划

**输出**：`docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`

**硬性门禁**：
```
在任何实现技能之前，必须先展示设计并获得用户批准。
这适用于每个项目，无论感知的简单程度如何。
```

---

#### 2. Writing Plans（编写计划）📝
**触发条件**：在拥有规范或需求的多步任务之前，在接触代码之前

**核心流程**：
- 范围检查（如果规范涵盖多个独立子系统，应该分解）
- 文件结构映射
- 编写细粒度任务（每个步骤 2-5 分钟）
- 计划文档审查循环

**输出**：`docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`

**任务粒度**：
```
每个步骤是一个动作（2-5 分钟）：
- "编写失败的测试" - 步骤
- "运行它以确保失败" - 步骤
- "编写最小代码使其通过" - 步骤
- "运行测试并确保通过" - 步骤
- "提交" - 步骤
```

---

#### 3. Test-Driven Development（测试驱动开发）🧪
**触发条件**：实现任何功能或错误修复，在编写实现代码之前

**铁律**：
```
没有失败的测试，就没有生产代码
```

**RED-GREEN-REFACTOR 循环**：

1. **RED - 编写失败的测试**
   - 编写一个显示应该发生什么的测试
   - 要求：一种行为，清晰的名称，真实代码

2. **验证 RED - 看它失败**
   - **强制执行。不要跳过。**
   - 确认：测试失败（不是错误），失败消息符合预期，失败是因为功能缺失（不是拼写错误）

3. **GREEN - 最小代码**
   - 编写最简单的代码来通过测试
   - 不要添加功能，重构其他代码，或"改进"超出测试

4. **验证 GREEN - 看它通过**
   - **强制执行。**
   - 确认：测试通过，其他测试仍然通过，输出干净（没有错误、警告）

5. **REFACTOR - 清理**
   - 只在绿色之后：删除重复，改进名称，提取助手
   - 保持测试绿色。不要添加行为

---

#### 4. Requesting Code Review（请求代码审查）👀
**触发条件**：任务之间，自动激活

**核心流程**：
- 审查计划
- 按严重程度报告问题
- 关键问题阻止进展

**审查重点**：
- 是否符合规范？
- 代码质量如何？
- 测试是否充分？
- 是否遵循了 DRY、YAGNI 原则？

---

#### 5. Subagent-Driven Development（子代理驱动开发）🤖
**触发条件**：拥有计划，如果框架支持子代理（Claude Code 等）

**核心流程**：
- 每个任务使用新的子代理
- 两阶段审查（规范符合性，然后代码质量）
- 检查和继续

**优势**：
- Claude 可以自主工作数小时而不偏离计划
- 每个子代理都是"热情的初级工程师"
- 完全隔离的工作空间

---

#### 6. Using Git Worktrees（使用 Git 工作树）🌳
**触发条件**：设计批准后，在实施之前

**核心流程**：
- 创建隔离的工作空间在新分支
- 运行项目设置
- 验证干净的测试基线

**优势**：
- 并行开发分支
- 隔离的工作环境
- 干净的提交历史

---

#### 7. Finishing a Development Branch（完成开发分支）✅
**触发条件**：任务完成时

**核心流程**：
- 验证测试
- 展示选项（合并/PR/保留/丢弃）
- 清理工作树

---

### 辅助技能

#### Systematic Debugging（系统性调试）🔍
**触发条件**：调试问题时

**4 阶段根因流程**：
1. 根因追踪
2. 深度防御
3. 基于条件的等待技术
4. 完成前验证

---

#### Verification Before Completion（完成前验证）✓
**触发条件**：声称修复完成之前

**核心问题**：它真的被修复了吗？

## 最佳实践 ⭐

### 开发任务最佳实践 ✅

- 每个项目都通过设计流程（即使是简单的）
- 一次问一个问题
- 提出多种方案并讨论权衡
- 编写细粒度任务（2-5 分钟）
- TDD：先写测试，看它失败，编写最小代码
- 在计划中包含确切的文件路径、完整代码、验证步骤
- 频繁提交
- 代码审查在任务之间

### 泛用式任务最佳实践 ✅

- 先评估任务复杂度，再决定使用哪种流程
- 批量操作优于循环操作，减少往返次数
- 并行处理独立任务，提升 token 效率
- 一次性返回完整结果，避免分步提示
- 优先使用已有信息（memory、文件、缓存）
- 选择最直接的工具完成任务
- 复杂任务生成结构化输出（文档、报告）
- 定期汇报进度，获取用户反馈
- 每步操作后验证结果，避免错误累积

### Token 效率优化 ✅

- **批量原则**：一次性收集所有需要的信息
- **并行原则**：并行处理独立的子任务
- **结构化原则**：一次性返回完整结果
- **缓存原则**：优先使用已有信息
- **工具优化**：选择最直接的工具完成任务
- **避免冗余**：去除重复和不必要的内容

### 通用禁止事项 ❌

- 在没有设计的情况下实现功能
- 在没有评估复杂度的情况下直接回答复杂任务
- 一次问多个问题（除非用户明确要求）
- 跳过 TDD（开发任务）
- 逐步提示复杂结果（应该一次性返回）
- 重复查询相同信息
- 分步执行可以批量完成的任务
- 在测试之前编写代码（删除它，重新开始）
- 添加不必要的功能（YAGNI）
- 编写大而复杂的任务
- 跳过审查循环

## 常见陷阱 🚨

### 反模式："这太简单了，不需要设计"

每个项目都经历这个过程。待办事项列表、单功能实用程序、配置更改——所有这些。"简单"的项目是未检查的假设造成最多浪费的地方。设计可以很短（对于真正简单的项目只有几句话），但你**必须**展示它并获得批准。

### 常见合理化（以及为什么它们是错误的）

| 理由 | 现实 |
|------|------|
| "太简单了，不需要测试" | 简单的代码也会破坏。测试需要 30 秒。 |
| "我会在之后测试" | 立即通过的测试证明不了任何东西。 |
| "之后测试达到相同目标" | 测试后 = "这有什么作用？" 测试前 = "这应该有什么作用？" |
| "我已经手动测试了所有边缘情况" | 临时 ≠ 系统性。没有记录，无法重新运行。 |
| "删除 X 小时的工作是浪费的" | 沉没成本谬误。保留未验证的代码是技术债务。 |
| "作为参考保留，先编写测试" | 你会适应它。那是测试后。删除意味着删除。 |
| "需要先探索" | 可以。丢弃探索，从 TDD 开始。 |
| "测试很难 = 设计不清楚" | 听从测试。很难测试 = 很难使用。 |
| "TDD 会让我变慢" | TDD 比调试更快。实用 = 测试优先。 |
| "手动测试更快" | 手动不证明边缘情况。每次更改都要重新测试。 |
| "现有代码没有测试" | 你在改进它。为现有代码添加测试。 |
