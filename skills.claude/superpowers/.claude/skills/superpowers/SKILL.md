# Superpowers — 泛用式 AI 任务处理框架

Superpowers 将 AI 代理转变为真正具备问题解决能力的伙伴。基于严格的 TDD 实践和系统化方法论，覆盖软件开发和各类复杂任务。

## 核心原则

1. **测试驱动开发 (TDD)** — 没有失败的测试，就没有生产代码
2. **系统性方法** — 系统化拆解，告别临时性思考
3. **复杂性降低** — 简单是首要目标
4. **证据优于声明** — 验证后再说成功
5. **效率优先** — 最大化 token 效率，批量处理，并行执行

## Phase 0：任务复杂度评估

收到请求后，首先判断任务类型：

**复杂任务**（必须启动 Superpowers）：
- 需要多步推理 / 多轮信息收集
- 涉及多个信息源或工具调用
- 需要创造性思维或方案设计
- 涉及权衡决策或不确定性
- 需要结构化输出（报告 / 文档 / 计划）

**简单任务**（直接回答）：
- 单一信息查询
- 单步文件操作
- 直接答案问题

**自动触发规则**：
- ✅ "帮我创建 / 分析 / 设计 / 优化 / 解决 / 制定..."
- ✅ 涉及多个步骤或子任务
- ❌ 单一查询 / 直接答案 / 单步操作

## 泛用式工作流（非开发任务）

### 流程 A：通用头脑风暴
**适用**：深度思考 / 设计 / 规划 / 分析

1. **理解任务** — 明确目标、约束、成功标准
2. **澄清问题** — 一次一个问题，确保理解准确
3. **探索方案** — 提出 2-3 种方法及权衡
4. **分块展示** — 分块呈现复杂结果，每块获取确认
5. **结构化输出** — 生成文档 / 报告
6. **验证确认** — 确认符合需求

**输出**：`docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`

### 流程 B：任务拆解与执行
**适用**：多步骤执行任务

1. **拆解** — 拆为 2-5 分钟的原子任务
2. **排序** — 按依赖关系和重要性排序
3. **执行** — 逐步执行，每步验证
4. **汇报** — 定期反馈进度
5. **汇总** — 汇总所有子任务结果

### 流程 C：信息收集与综合
**适用**：多信息源收集任务

1. **需求分析** — 明确需要什么、为什么需要
2. **信息源识别** — 确定最佳来源（搜索 / 文件 / API）
3. **并行收集** — 尽可能并行以节省 token
4. **综合** — 去重、验证
5. **结构化输出** — 报告 / 文档

### 流程 D：方案设计与决策
**适用**：设计解决方案或做决策

1. **需求分析** — 理解问题本质和约束
2. **方案生成** — 2-3 个可行方案
3. **方案评估** — 技术可行性、成本、风险、可维护性
4. **推荐方案** — 推荐 + 理由
5. **实施计划** — 为推荐方案制定计划

## 开发任务工作流

```
需求 → Brainstorming → Design Spec → Writing Plans → TDD 实现 → Code Review → 完成分支
```

### 1. Brainstorming（头脑风暴）
**触发**：任何创意工作之前

- 探索上下文，询问澄清问题（一次一个）
- 提出 2-3 种方案及权衡
- 分块展示设计，每部分获取批准
- 输出：`docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`

**硬性门禁**：实现任何技能前，必须先展示设计并获得用户批准。

### 2. Writing Plans（编写计划）
**触发**：拥有规范/需求，在接触代码之前

- 文件结构映射，细粒度任务（每步 2-5 分钟）
- 每步有明确输入、输出和验证命令
- 输出：`docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`

### 3. Test-Driven Development（TDD）
**触发**：实现任何功能或 Bug 修复之前

**铁律**：没有失败的测试，就没有生产代码

**RED-GREEN-REFACTOR**：
1. **RED** — 编写一个失败的测试（一个行为，清晰名称）
2. **验证 RED** — 强制运行，确认失败（不是错误）
3. **GREEN** — 编写最简代码使测试通过，不添加额外功能
4. **验证 GREEN** — 强制运行，确认通过且其他测试未破坏
5. **REFACTOR** — 仅在绿色后清理，保持测试绿色

### 4. Requesting Code Review
**触发**：任务之间，自动激活

- 按严重程度报告（Critical / High / Medium / Low）
- Critical 问题阻止进展
- 重点：规范符合性、代码质量、TDD、DRY/YAGNI

### 5. Subagent-Driven Development
**触发**：有计划且框架支持子代理

- 每个任务使用新子代理，两阶段审查（规范符合性 → 代码质量）

### 6. Using Git Worktrees
**触发**：设计批准后，实施之前

- 在新分支创建隔离工作空间，验证干净基线

### 7. Finishing a Development Branch
**触发**：任务完成时

- 验证测试，展示选项（merge / PR / keep / discard），清理工作树

## Token 效率优化

| 原则 | 做法 |
|---|---|
| 批量操作 | 一次性收集所有需要的信息 |
| 并行处理 | 独立任务并行执行 |
| 结构化输出 | 一次性返回完整结果，不逐步提示 |
| 缓存利用 | 优先使用已有信息（memory / 文件 / 缓存） |
| 工具优化 | 选择最直接的工具，避免不必要的中间步骤 |
